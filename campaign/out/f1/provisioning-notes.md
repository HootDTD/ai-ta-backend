## What happened

`campaign.cast.teacher.provision_authored` was driven against the real
`POST /apollo/authored-sets` -> poll -> approve path with the checked-in
fixture PDFs (`campaign/cast/materials/linear_motion_{problem,solution}.pdf`,
generated by `campaign/cast/materials/generate_fixtures.py`).

**Attempt 1 (set_id=1, original fixture text)** — 0/2 promoted:

- Problem 1(a): `gate 6: malformed equation in 'eq2': KG entry 'eq2' has
  malformed equation 'v = 0 + (2.0)(5.0) = 10.0 m/s': expected exactly one
  '=' but found 2`
- Problem 1(b): `gate 6: malformed equation in 'eq2': KG entry 'eq2' has
  malformed equation 'x = v0*t + (1/2)*a*t^2 = 0 + 0.5*(2.0)*(5.0)^2 = 25.0
  m': unsupported operand type(s) for ^: 'Add' and 'Integer'`

**Finding A (real defect, not fixed in this task):** the fixture's original
solution text used the extremely common physics-writeup style of a chained
equality (`symbolic = numeric-substitution = final-answer` all on one line)
and caret (`^`) for exponentiation. Gate 6's equation parser requires
exactly one `=` per equation string and does not accept `^` (sympy parses
`^` as XOR, not power, producing `unsupported operand type(s) for ^`). Both
are mainstream textbook/worked-solution notations that a real teacher PDF
is very likely to contain — this is a genuine WU-AAS ingestion gap, not a
fixture-authoring mistake. Reported here rather than silently avoided.

**Remediation attempted (in scope: the fixture is campaign test material,
not production code)** — rewrote `LINEAR_MOTION_SOLUTION_TEXT` in
`generate_fixtures.py` to (1) split each equation onto its own line with a
single `=`, prose-stating the numeric result on the following line instead
of chaining it, and (2) use `**` instead of `^`. Regenerated the PDFs via
`python -m campaign.cast.materials.generate_fixtures`.

**Attempt 2 (set_id=2)** — 1/2 promoted:
- Problem 1(a) (`v = v0 + a*t`): **promoted**, `concept_problem_id=11`.
- Problem 1(b) (`x = v0*t + (1/2)*a*t**2`): rejected —
  `gate 4: foreign symbol 'x' is not canonical and not normalizable (the
  sole foreign-symbol guard)`.

**Finding B (real defect):** the canonical-symbol registry does not
recognize `x` for position/displacement even though it is the single most
standard kinematics symbol. Renamed the fixture's distance variable to `d`
as a remediation (still a legitimate, if less conventional, choice a real
teacher could make) rather than special-casing `x` in the parser.

**Attempt 3 (set_id=3, `d` instead of `x`)** — 1/2 promoted:
- Problem 1(a): rejected as `gate 8: duplicate problem (dup hash ... already
  exists)` — correctly recognized as the same problem already promoted in
  set_id=2.
- Problem 1(b) (`d = v0*t + (1/2)*a*t**2`): **promoted**,
  `concept_problem_id=12`.

**Net result:** both parts of the fixture problem are now promoted
(`concept_problem_id` 11 and 12), both attributed to the SAME real minted
concept, `apollo_concepts.id=5` (`slug='linear-motion'`, filed under the
`fluid_mechanics` **subject**, not a new `linear_motion` subject — the
WU-AAS pipeline infers subject/concept slugs itself and does not create a
subject per campaign registry key; **Finding C**, a naming-surface mismatch
between `campaign.cast.subjects.LINEAR_MOTION` and what the live pipeline
actually files things under). 37 `apollo_kg_entities` rows exist under
`concept_id=5`, and the same 37 nodes are projected to Neo4j `:Canon`
(`concept_id: 5`) automatically by `run_authored_set_provisioning` — no
separate `seed_canon_projection` step was needed for the WU-AAS path.

**Finding D (data-quality defect in the mint):** because the two problems
were ingested as two SEPARATE `POST /apollo/authored-sets` uploads against
the same concept, the entity-extraction pass ran twice independently with
no cross-run deduplication against already-minted entities of the same
concept. The result is heavy duplication under different synthetic keys:
`eq.eq_motion` / `eq.eq_velocity_formula` / `eq.eq1` / `eq.eq2` all
represent the same v = v0 + a*t equation family; `def.def1..def4` duplicate
`def.def_acceleration` / `def.def_distance` / `def.def_initial_velocity` /
`def.def_time` / `def.def_velocity`; `varmap.var1..var4` /
`varmap.vm1..vm3` duplicate the named `varmap.var_a` / `var_t` / `var_v0` /
`var_x` entries; `simp.simp1` / `simp.simp2` duplicate
`simp.simp_calculate_distance` / `simp.simp_calculate_velocity`. 37 total
entities is roughly 2x what a single coherent 2-part kinematics problem
should mint (~15-18 meaningful nodes).

## Reconciliation against the 4 provisional persona files

`campaign/cast/personas/linear_motion/reference/kinematics_constant_acceleration/`
(the D2 hand-authored provisional reference, routed via
`validate.py::PROVISIONAL_SUBJECTS`) uses these keys:

| Provisional key | Real minted candidate(s) | Confidence |
|---|---|---|
| `cond.constant_acceleration` | `cond.cond_given_values` (closest condition; no entity explicitly encodes "acceleration is constant" as its own step) | low |
| `eq.kinematics_velocity` (v=v0+a*t) | `eq.eq_motion` **or** `eq.eq_velocity_formula` **or** `eq.eq1` (three near-duplicate candidates, Finding D) | ambiguous |
| `eq.kinematics_position` (x=v0*t+1/2*a*t^2) | `eq.eq2` is the only other equation slot, but its `display_name` ("Calculated final velocity") suggests it may actually be a velocity-family duplicate, not the position equation — no equation entity's `display_name` unambiguously says "position"/"distance" | unresolved |
| `proc.compute_final_velocity` | `proc.proc_substitute_values` / `proc.proc1` / `proc.ps1` (three candidates, no numeric target field to disambiguate) | ambiguous |
| `proc.compute_distance` | same pool as above | ambiguous |
| `misc.forgets_half_factor` / `misc.ignores_constant_acceleration` | **no misconception-kind entities exist under concept_id=5 at all** (0 rows with `kind='misconception'`) | not minted |

**Decision: did NOT rewrite the 4 persona files.** The real mint's
duplicate-entity defect (Finding D) makes an honest, verifiable 1:1
key mapping impossible without guessing — every equation/procedure/
definition role has 2-4 same-shape candidates with no reliable
disambiguating field (`payload->>'equation'` is empty for every row in this
mint; see the `eq` column, all NULL — a further data-quality gap, **Finding
E**: entities were minted without their symbolic equation payload
populated, even for `kind='equation'` rows). Writing a mapping under this
uncertainty would make `campaign.judges.s3_student_fidelity`'s
`ledger_vs_expected` diff silently compare against a fabricated ground
truth rather than an honestly-provisional one — worse than leaving the
files as clearly-labeled provisional data. `linear_motion` stays in
`validate.py::PROVISIONAL_SUBJECTS`; `test_persona_schema.py` was re-run
unmodified and still passes (nothing to break — no edits made).

**Recommended next step (not done in this task, out of scope for F1a):**
either (a) export the real minted `concept_id=5` rows to an
`apollo/subjects/linear_motion/` filesystem tree (mirroring the seeded
subjects' `AUTHORING.md` convention) and point `validate.py` at it, forcing
the disambiguation question to be answered by a human reading the real
`payload` JSON of each duplicate candidate, or (b) fix Finding D first
(dedup entities within a concept across sequential authored-set uploads)
so a future WU-AAS mint of the same fixture doesn't reproduce the
ambiguity, then re-mint and reconcile.
