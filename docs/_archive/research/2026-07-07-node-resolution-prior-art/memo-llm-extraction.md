# LLM-based structured extraction & LLM-as-matcher (2024–2026)

Research memo for the Apollo node-resolution recall problem (strong explanations resolving 1/5 nodes; edge coverage <=0.25; transcript-audit safety net over-credits). Angle: LLM KG extraction, LLM-as-judge checklist grading, structured-output reliability, multi-pass extraction, and over-credit mitigation.

## Key findings

1. **EDC — Extract, Define, Canonicalize (EMNLP 2024)** — https://arxiv.org/abs/2404.03868, code https://github.com/clear-nus/edc. Three-phase pipeline: open-extract triplets, have the LLM write a natural-language *definition* for each extracted relation/entity, then canonicalize against a target schema by embedding-similarity retrieval over definitions followed by LLM verification. Explicitly supports the "target schema exists" setting (ours). Their add-on "Schema Retriever" (EDC+R) is a trained component that retrieves relevant schema items *before* extraction to boost recall. This is architecturally our resolver, done as retrieve-candidates-then-verify instead of tiered exact/fuzzy/NLI gates.
2. **KGGen (2025)** — https://arxiv.org/abs/2502.09956, `pip install kg-gen`. Attacks node fragmentation/synonym proliferation with LLM-driven *clustering* of extracted entities into canonical nodes; introduces the MINE benchmark. Confirms that raw LLM extraction produces many surface variants of the same concept — exactly the derived-equation/paraphrase-variant failure we see — and that a dedicated cluster/canonicalize stage is the fix, not stricter extraction.
3. **GraphRAG "gleanings" (Microsoft, 2024)** — https://arxiv.org/html/2404.16130v2, repo https://github.com/microsoft/graphrag (MIT). Production KG extraction runs *multiple rounds* per chunk: after each pass, a forced yes/no (logit-bias 100) asks whether entities were missed; if yes, the continuation "MANY entities were missed in the last extraction" triggers another pass. Also quantifies chunk-size effect: GPT-4 extracted ~2x as many entity references at 600-token chunks vs 2400. Direct precedent for multi-pass, per-turn extraction from our transcripts.
4. **LangExtract (Google, 2025, actively maintained)** — https://github.com/google/langextract (Apache-2.0). Production IE library whose two headline features map 1:1 onto our defects: (a) `extraction_passes>1` — independent sequential extractions merged by union (first-wins on overlap) explicitly "to improve recall"; (b) *source grounding* — every extraction carries a `char_interval` aligned to the source (exact match, then LCS fuzzy alignment, threshold 0.75, statuses MATCH_EXACT/MATCH_LESSER/MATCH_FUZZY/None). Ungroundable extractions are flagged — the exact mechanism our over-crediting auditor lacks.
5. **Multi-sample union beats best-single-sample for recall.** Atomic Self-Consistency (EMNLP 2024, https://aclanthology.org/2024.emnlp-main.706.pdf) shows merging atomic subparts of multiple stochastic samples significantly beats selecting the best single generation (USC) on recall-sensitive QA; ~5 samples captures most of the gain. ThinkTwice (https://arxiv.org/html/2601.18395v2, code https://github.com/MikelZubi/ThinkTwice) shows greedy decoding systematically underperforms sample-then-select for document-level IE; plain majority voting fails because the modal extraction is empty. EVE (https://arxiv.org/html/2602.06103) gets +24% recall AND +29% precision by splitting generation into extract → validate → enumerate stages. LECITE (https://www.mdpi.com/1999-5903/18/1/32) ablates cleanly: iterative re-prompting with prior-round entities injected raises *recall*; consistency filtering against the source raises *precision* — separate levers.
6. **Checklist/per-criterion binary judging beats holistic scoring.** TICK (https://arxiv.org/abs/2410.03608): decomposing an instruction into LLM-generated YES/NO checklist questions raises exact judge–human agreement 46.4% → 52.2% vs direct scoring. CheckEval (EMNLP 2025, https://arxiv.org/abs/2403.18771, code https://github.com/yukyunglee/CheckEval): Boolean sub-questions improve cross-evaluator agreement by +0.45 and cut score variance, with per-item traceability. For us: the transcript audit should be N independent per-node binary judgments ("did the student teach X? quote the turn"), never one holistic coverage judgment.
7. **Reference-grounding the judge slashes over-credit — with a known side effect.** LLM-as-judge validity in physics assessment (https://arxiv.org/html/2603.14732): under blind marking, 13 exam questions had *all five* frontier models (GPT-5.2, Claude Opus 4.5, Gemini Pro 3, DeepSeek-V3.2, Grok 4.1) award full marks where humans gave zero. Supplying the official solution cut over-credit cases 13 → 2 (−85%) but raised under-credit 9 → 12 (rigid exact-matching), and with deliberately corrupted references all models deferred to the wrong reference. Also: judge validity tracks rubric granularity, not model capability — rank agreement ρ>0.6 for structured questions vs ρ≈0.1 for essays. MT-Bench (https://arxiv.org/abs/2306.05685) found the same qualitative effect earlier: reference-guided judging fixes most math-grading failures.
8. **LLM-as-matcher has sharp caveats.** LLMAEL (https://arxiv.org/pdf/2407.04020): using an LLM to re-rank entity candidates *underperformed* the specialized retriever alone (AIDA F1: BLINK 82.01 vs LLM re-rank-100 70.95); LLMs work better as context augmenters than as final matchers. AGNUS (IJCNLP 2025, https://aclanthology.org/2025.ijcnlp-long.174.pdf): candidate *order* alone moves F1 by up to 15.2%; constrained decoding restricted to the candidate set eliminates hallucinated matches. GenDecider (NAACL 2024, https://aclanthology.org/2024.naacl-short.22.pdf, code https://github.com/kangISU/GenDecider): re-rankers must support an explicit "None of the Candidates" output — forced choice inflates false positives. ARTER (EMNLP-industry 2025, https://arxiv.org/pdf/2510.20098): route easy mentions to a cheap matcher and only hard cases to LLM reasoning (multiple-choice over top-30 candidates, "-1 if none fits"); few-shot + CoT beat contrastive and ReAct prompting; ~2x cheaper than LLM-on-everything at equal accuracy.
9. **Decompose-then-verify as candidate generation.** FActScore (https://arxiv.org/abs/2305.14251) and VeriScore (EMNLP Findings 2024, https://arxiv.org/abs/2406.19276) split long text into atomic claims and verify each independently; VeriScore adds "extract only *verifiable* claims" and pronoun decontextualization. For us: parse the transcript into decontextualized atomic claims first (recall-friendly, cheap), then match each claim against reference nodes — instead of asking the parser to emit node-shaped objects directly.
10. **Structured-output reliability is a format problem, not a recall lever — and it's contested.** "Let Me Speak Freely?" (EMNLP-industry 2024, https://aclanthology.org/2024.emnlp-industry.91.pdf) reports reasoning degradation under format restriction; dottxt's rebuttal (https://blog.dottxt.ai/say-what-you-mean.html) shows properly-done constrained decoding matches or beats unstructured. JSONSchemaBench (https://arxiv.org/abs/2501.10868) benchmarks engines (Guidance/Outlines/XGrammar/OpenAI/Gemini) on 10K real schemas; the Structured Output Benchmark (https://arxiv.org/pdf/2604.25359) shows 99.97% JSON-validity can coexist with 0.486 content-correctness. Lesson: schema-valid parser output tells us nothing about extraction recall; measure content recall separately.
11. **Dialogue-specific KG extraction is thin.** Closest work is dialogue knowledge tracing (LAK 2025, https://learninganalytics.upenn.edu/ryanbaker/Dialogue_KT_LAK_25-2.pdf; difficulty-aware follow-up https://arxiv.org/pdf/2605.01097) — LLMs label which KCs each tutoring turn evidences. No mature "KG-extraction-from-tutoring-dialogue" system exists; document-IE techniques + per-turn chunking is the transferable state of the art. Honest gap: nobody has published our exact task (grading a *student-as-teacher* transcript against a reference concept graph).

## Adoptable artifacts

- **google/langextract** — Python, Apache-2.0, very active (2025). Adopt as parser front-end: per-turn extraction with `extraction_passes=3`, char-interval grounding; its alignment statuses give us machine-checkable evidence spans for every candidate, and its WordAligner (exact→LCS fuzzy, density gate) is a better-engineered version of our fuzzy tier.
- **clear-nus/edc** — research code for EDC (EMNLP 2024). Port the *pattern*, not the repo: write affirmative NL definitions for every reference node (we already have NLI labels ≈ these), embed them, retrieve top-k nodes per extracted claim, LLM-verify each (claim, node-definition) pair. Their Schema Retriever shows a trained retriever specifically lifts recall.
- **kg-gen (pip)** — Stanford, MIT-licensed per repo. Its entity-clustering stage is adoptable for collapsing student paraphrase variants before resolution (cluster candidates, resolve cluster representatives).
- **microsoft/graphrag** — MIT. Copy the gleaning loop prompt verbatim (forced yes/no "were entities missed?", "MANY entities were missed" continuation) into our parser.
- **yukyunglee/CheckEval** — checklist-construction scaffolding for rebuilding the transcript audit as per-node Boolean questions.
- **kangISU/GenDecider + ARTER (paper only)** — design references for the resolver-as-matcher: multiple-choice over retrieved candidate nodes with an explicit abstain option; route only unresolved candidates to the expensive LLM matcher.

## Recall lessons

- **Recall comes from candidate generation; precision from verification.** Every successful system (EDC, EVE, LECITE, ASC, VeriScore) extracts liberally then filters with a grounded check. Our precision-first NLI thresholds sit at the wrong stage: loosen candidate admission, tighten span-grounded verification.
- **Run K passes and union.** LangExtract `extraction_passes`, GraphRAG gleanings, ASC (5 samples captures most gain), ThinkTwice. Union-merge with dedupe by canonical key; do NOT majority-vote (modal output is under-extraction).
- **Chunk small.** GPT-4 extracted ~2x entity references at 600 vs 2400 tokens (GraphRAG). Extract per dialogue turn (or turn-pair), not per transcript.
- **Give the matcher the reference schema as explicit candidates** (multiple-choice + abstain), never open generation against canonical keys; constrain output to the candidate set; randomize/neutralize candidate order (up to 15.2% F1 order sensitivity).
- **Definitions are the anchor representation.** Match against per-node affirmative NL definitions (EDC; our NLI labels), embedded for retrieval — not against key strings or hand aliases.
- **Decontextualize claims before matching** (VeriScore): resolve pronouns/ellipsis from dialogue context, or matching fails on surface grounds.
- **For the auditor: reference-grounded, span-quoting, per-node binary.** Grounding cut over-credit 85%; binary checklists raise agreement; require a verbatim quote per YES and programmatically verify the quote exists in the transcript (LangExtract-style alignment) — an unverifiable quote is an auto-NO. Expect and monitor the under-credit side effect.
- **Use the audited LLM pass as a recall *candidate generator* feeding the deterministic resolver, not as a score source** — consistent with the physics-paper finding that judges defer to whatever reference they're handed.

## Dead ends

- **Holistic transcript-level LLM scoring** — essay-style judging shows near-zero rank agreement (ρ≈0.1) and unanimous-over-credit failure modes across all frontier models; unfixable by model upgrades since validity tracks task structure.
- **Naive LLM-as-reranker over the full candidate list** — LLMAEL's measured regression (82.01→70.95 F1) plus order bias; use routing + constrained choice + abstention or keep the specialized matcher.
- **Constrained JSON decoding as a recall fix** — it guarantees parse validity, not content coverage (99.97% valid vs 48.6% correct); worth having for pipeline robustness only.
- **Single-best-sample self-consistency (USC-style voting)** — intersection/mode semantics *lowers* extraction recall; only union/merge variants help.
- **Fine-tuning a bespoke extractor now** — LECITE's LoRA gains are marginal (F1 0.358 vs 0.349 strict) for substantial data/ops cost; prompt-level multi-pass + retrieval-verify is the cheaper first move.
- **Hand-authored alias lists** — the literature's replacement is embedding retrieval over definitions + clustering (EDC, KGGen); aliases don't generalize to derived/paraphrased forms (matches our internal Phase-1b rejection).
