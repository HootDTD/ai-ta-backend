## 0. What WU-5B is (and the lines it must not cross)

WU-5B consumes the FROZEN WU-5A path. Three concerns, three very different shapes:

- **DECAY** is ~30 lines of pure arithmetic + ONE injected call into the §3 fold. Smallest, safest concern — a convex combination cannot break the never-0 invariant; it *repairs* zeros. **v1-IN.**
- **RQ5 HEDGE** is a small multiplier (~25 lines) BUT hides a real keyspace bridge (negotiation moves are keyed by Neo4j `entry_id`, the §3 fold groups on `entity_id`), a 3-move-type-to-one-multiplier ambiguity, AND a **spec-deviation** (the literal §3 "multiply per evidence item" rule lets a negotiation move partially CANCEL a same-Done misconception — reproduced −0.044). **v1-IN but reduced** (persist `negotiation_move` + apply the multiplier under a gated flag), with the keyspace bridge pinned and the suppression escalated to the orchestrator (it OVERRIDES the spec).
- **JANITOR** is the LARGEST concern — NOT "drain a flag." The shadow `ShadowGradeResult` is ephemeral (the persisted findings rows are a lossy one-way projection), so the janitor must **re-run `run_graph_simulation` from resolution** = 2 LLM calls + Neo4j + full input re-assembly the live `done.py` does inline — INCLUDING `old_rubric`, a hard-required input the v1 draft dropped. It also needs a poison-row dead-letter, per-row session discipline, a LAYER3-flag interlock, and backoff pacing. The one concern with a genuine v1-vs-defer call.

WU-5B owns NO grading algorithm, NO findings production, NO run/findings persistence, NO new belief math beyond the decay blend + the hedge multiplier. It does NOT wire `opposes`-map population (the structurally-empty `opposes_map` is a SEPARATE pre-LIVE follow-up, §8).

## 2. Recommendation (the split)

**Split WU-5B into THREE sub-units, build in this order:**

1. **WU-5B1 — DECAY** (pure `decay.py` + injected `prior_transform` into the §3 fold, with the dt-hoist). Smallest, safest, no migration, no infra. Ships FIRST and de-risks the injected-parameter seam (OPEN-DECISION #1) the hedge also depends on.
2. **WU-5B2 — RQ5 NEGOTIATION HEDGE** (the `{node_id: move}` read + the entity-keyed multiplier + persist `negotiation_move`). Depends on WU-5B1's injected-parameter seam. No migration, no infra beyond real-PG for the persist round-trip.
3. **WU-5B3 — RETRY JANITOR** (the worker-loop drain + re-run-from-resolution + clear-flag + dead-letter + backoff). Largest, riskiest, the ONLY concern touching the worker + Neo4j + LLM. Migration 028. Depends on WU-5B1+5B2 because the janitor's `run_learner_update` call must **inherit the same decay+hedge FLAGS** the live path used so its belief is computed by the same rules (NOT "byte-identical" — the re-run re-calls the live LLM adjudicator/auditor, so findings/events can legitimately differ per §3 L462-464; the determinism guarantee is scoped to the Δt anchor = original `done_ts`, never `now()`, `…architecture-decision.md:417`).

**Why three, not one or two:** orthogonal infra footprints (pure / real-PG / worker+Neo4j+LLM), orthogonal risk (trivial / medium-ambiguity / large-rerun), orthogonal coverage gates (golden-vector / round-trip / worker-loop integration). The build order is **dependency-forced:** the janitor must call decay+hedge with the live flags to grade by the same rules, so it lands LAST.

**The first unit is WU-5B1 (DECAY).** Pins the injected-parameter seam (+ the dt-hoist), zero infra, exhaustively golden-vector-testable.

### The v1-vs-defer call per concern (OPINIONATED)

- **DECAY → v1-IN.** Trivial, safe, fully plumbed. Load-bearing for the TRANSITION of a re-seen entity (correctly forgetting the gap before fresh evidence), NOT for the unseen-entity readout (that needs read-time decay, deferred). No reason to defer the update-time blend.
- **RQ5 HEDGE → v1-IN but REDUCED to the negotiation-move multiplier ONLY.** The §10 chat-keyword hedge (persisting `extract_and_filter_keywords` output as a JSONB column on chat turns, `…architecture-decision.md:1461-1466`) is a Hoot-CHAT-side `/ask`-path change, NOT an Apollo learner-model-fold change — OUT of WU-5B (defer to a Hoot-chat unit). The multiplier ships behind a flag (default OFF) gated on calibration. **Spec-text divergence to surface (OPEN-DECISION #4):** §12 phase-5 (L1534) LITERALLY names "RQ5 hedge (persist chat keywords)" as the phase-5 RQ5 deliverable — i.e. the §10 chat-keyword persist. This proposal RE-SCOPES phase-5 RQ5 to the §3 negotiation multiplier (the fold-side hedge WU-5B already owns) and defers the chat-keyword persist. That re-reading is defensible (the multiplier touches the fold; the chat persist is on the `/ask` path) but contradicts the literal phase-5 line — confirm with the orchestrator.
- **JANITOR → v1-IN but this is the genuine judgment call (OPEN-DECISION #2).** The worker-loop drain is the right design but the heaviest unit. A defensible LIGHTER v1 is **opportunistic-on-next-session ONLY** (drain a student's own pending attempts when they start their next Done), deferring the worker-loop sweep. Trade: opportunistic-only leaves a never-returning student's mastery stale forever and injects 2 LLM calls into session-init. RECOMMENDATION: ship the **worker-loop as primary + opportunistic as a cheap backstop** (the worker process exists; bounds drain latency) — but flag the sync-upload-loop vs async-Apollo-loop interleaving as the principal risk and let the orchestrator confirm scope before building 5B3.

A two-way split (decay+hedge / janitor) was considered: rejected because decay and hedge have independent flags, independent tests, and independent spec ambiguities (the hedge's keyspace bridge + the spec-deviating suppression is a whole research question the decay unit must not carry).

## 3. Sub-units

### WU-5B1 — between-session decay (§3 Step 0)

**One-line:** Decay the recompute-base prior toward `COLD_START_PRIOR` by `w = 1 − e^(−k·Δt_days)` (`k=0.05`) BEFORE the Bayes step, at UPDATE time, Δt anchored to the frozen `done_ts` (never `now()`), clamped `Δt ≥ 0`.

**SEAM (owns vs 5B2/5B3):** WU-5B1 owns the decay arithmetic + the dt-hoist + the ONE injected call into the §3 fold. It does NOT touch negotiation, the janitor, or any read path. Handoff: a pure `decay_toward_prior` function + a `prior_transform` callable threaded into `persist_learner_update`.

**Scope — files NEW:**
- `apollo/learner_model/decay.py` (NEW) — pure, mirrors `belief.py` (named constants, `import math`, no IO):
  - `DECAY_K: float = 0.05` (§3 L413; half-life `ln2/k = 13.86` d).
  - `decay_weight(dt_days: float, k: float = DECAY_K) -> float` — `1 − e^(−k·max(0.0, dt_days))` (the `max(0,·)` is the negative-Δt clamp, see key_risks).
  - `decay_toward_prior(belief, prior, dt_days, k=DECAY_K) -> tuple[float,float,float]` — `(1−w)·belief + w·prior`. PURE convex combination; NO renormalize needed (a convex combo of two sum-1 vectors is sum-1 — reproduced exactly).

**Scope — files EDIT (frozen — via injected-parameter extension + the dt-hoist, OPEN-DECISION #1):**
